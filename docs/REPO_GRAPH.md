# Repository graph and context packs

Hermes Legion Commander builds a local Graphify-style repository knowledge graph before worker prompts are sent to Codex CLI or Claude Code. The graph is designed to cut supervisor navigation time and prompt token volume by turning repository structure into durable, queryable context.

## Design goals

- Navigate by structure before raw search.
- Keep source analysis local by default.
- Preserve provenance: `EXTRACTED`, `INFERRED`, and `AMBIGUOUS` edges are visibly different.
- Reuse cached parse results when files do not change.
- Produce both human-readable reports and machine-readable graph data.
- Generate task-specific context packs small enough to inject into worker prompts.

## Outputs

Each run receives these files under `shared-context/`:

- `repo-map/graph.json` — persistent graph with file, symbol, module, call, entrypoint, and community nodes; extracted/inferred edges; confidence values; source path evidence; language counts; kind counts; hotspots; surprising cross-connections; and knowledge gaps.
- `repo-map/graph.html` — local interactive browser view with search, type filters, selected-node details, neighbors, and hotspots.
- `repo-map/GRAPH_REPORT.md` — Graphify-compatible report name for assistant workflows.
- `repo-map/REPO_MAP.md` — Commander navigation report with the same content as `GRAPH_REPORT.md`.
- `repo-map/repo-map-index.jsonl` — one indexed row per file for cheap lookup.
- `repo-map/cache/` — SHA-256/parser-version keyed per-file parse cache.
- `repo-map/wiki/` — one markdown page per community and indexed file for Obsidian-style navigation.
- `repo-context-pack.md` — task-specific start-here pack generated from the current stage prompt.
- `repo-context-pack.json` — structured version of the task-specific file selection.

These files are generated by local static analysis. Source files are read locally and no external graph service or LLM semantic extractor is required.

## Extracted facts

The local graph extracts:

- Python imports, local import resolution, classes, functions, methods, signatures, decorators, base classes, docstrings, rough complexity, and intra-file call edges.
- JavaScript and TypeScript imports, relative import resolution, functions, classes, and approximate call sites.
- Go, Rust, Java, Kotlin, and C# import/type/function symbols from regex-backed static scans.
- Markdown headings, links, code fences, and path references.
- TOML sections, Python project entrypoints, and dependency names.
- JSON/YAML keys and path references.
- SQL objects and table/reference edges.
- Shell/PowerShell function and source/include references.
- Dockerfile instructions and base images.
- PDF/image/audio/video assets as local metadata nodes without OCR or cloud upload.

## Communities and hotspots

Commander builds structural communities from local file-to-file relationships and directory structure. Each file receives a `community_id`, while `GRAPH_REPORT.md` lists community summaries, god-node/hotspot files, surprising cross-community connections, and knowledge gaps such as missing tests or undocumented central Python modules.

The centrality score is intentionally a navigation heuristic. It combines incoming/outgoing graph relationships, symbol density, entrypoint status, and special weight for config/script/schema files. It is not proof that a file should be edited.

## Query and path traversal

The package includes a repo-graph command group:

```powershell
hermes-legion-commander repo-graph build . --out shared-context/repo-map --task "fix login retry tests"
hermes-legion-commander repo-graph query "login retry" --graph shared-context/repo-map/graph.json --budget 20
hermes-legion-commander repo-graph path "login" "database" --graph shared-context/repo-map/graph.json --max-depth 5
```

Workers should use query results and path traversal before broad recursive reads. This allows Codex CLI and Claude Code to open likely paths and graph neighbors first.

## How routing uses it

Scope assessment records `repo_facts`, including indexed file count, language counts, file-kind counts, multimodal asset counts, and truncation status. Large repositories raise the estimated scope before learned routing is applied, while low-risk tasks in small repositories can still be downgraded to low effort when prior ledger data supports that.

## How workers should use it

Workers are instructed to read `repo-context-pack.md` and `repo-map/REPO_MAP.md` before broad repository search. The context pack selects likely files from task terms, symbols, imports, tests, docs, config, schema, entrypoints, communities, and graph neighbors. Workers should open those files first, then expand only when imports, failing tests, path edges, or diffs prove additional files are relevant.

## Difference from full Graphify

This implementation now matches the main Graphify workflow shape: persistent graph JSON, graph report, interactive HTML, cache, community structure, wiki pages, query/path traversal, confidence/provenance, and graph-first assistant context. It still does not claim feature parity with Graphify's optional LLM semantic extraction, Tree-sitter coverage across dozens of languages, full multi-modal content understanding, Leiden clustering, or MCP/tool-hook integrations. Those can be layered later behind explicit opt-in provider settings.


## All modes

The repository graph is refreshed by `ensure_shared_context` and prompt assembly, so council, checkpoint competition, cross-validation, final verification, dry-run, and Hermes supervisor prompts all receive repo-map facts or context-pack guidance. Workers should read `repo-context-pack.md`, `repo-map/REPO_MAP.md`, or `repo-map/GRAPH_REPORT.md` before broad repository traversal.
